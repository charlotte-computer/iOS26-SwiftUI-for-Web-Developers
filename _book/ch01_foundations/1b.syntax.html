3  Swift Syntax & Lexical Rules

Welcome to the lexical breakdown of Swift. When transitioning from JavaScript, you will notice Swift is structurally cleaner, strictly enforced by the compiler, and relies heavily on minimizing visual clutter.

3.1 The Rules of Syntax

Here is a fast, factual rundown of the core lexical rules in Swift.

1. Semicolons are Dead

Fact: Swift does not require semicolons ; at the end of statements.

  • In modern JavaScript, semicolons are technically optional but are widely enforced by linters (like Prettier/ESLint) to prevent interpreter errors.
  • In Swift, semicolons are genuinely discouraged by the compiler.
  • Rule: Only use a semicolon if you are writing two completely separate statements on the exact same line (which you should rarely ever do).
// Correct Swift
let name = "Charlotte"
let age = 28

// Discouraged Swift (Do not do this)
let name = "Charlotte"; let age = 28;

2. Variables and Constants (let vs var)

Fact: let in Swift is functionally identical to const in JavaScript. var in Swift is identical to let in JavaScript.

  • Rule: Always default to using let when declaring new data. Only change it to var if the Swift compiler explicitly throws an error telling you that the value needs to mutate later.
NoteWhat does this look like in JavaScript? 
const name = "Charlotte" // Immutable
let age = 28 // Mutable
let name = "Charlotte" // Immutable (Use this default)
var age = 28 // Mutable (Use only if changing)

Variables have a type which is established when the variable is declared and cannot be changed later.

The type of the variable is derived from the first value that is assigned to it. For example, let name = "Charlotte" is a String, and let age = 28 is an Int.

3. String Interpolation

Fact: Swift uses \() to inject variables into strings. - In JavaScript, you switch to backticks and use ${variable}. - In Swift, you keep standard double quotes "" and inject the variable natively using a backslash and parentheses \(variable).

5. Parentheses in Control Flow

Fact: Swift deliberately drops the parentheses () around if, guard, and switch statements. - Rule: The curly braces {} encompassing the block are strictly mandatory, but the parentheses holding the boolean condition are entirely omitted to reduce visual noise.

NoteWhat does this look like in JavaScript? 
if (age > 18) {
    console.log("Adult");
}

Swift Equivalent:

// ⬇️ No parenthesis needed around 'age > 18'
if age > 18 { 
    print("Adult")
}

6. Multi-Line Dot Syntax (Modifier Chaining)

Fact: Swift heavily relies on chaining methods together using dot . notation. For readability in enterprise applications, these chains are almost always broken across multiple vertical lines. - Rule: The dot . must start on the new line, typically aligned directly beneath the object it modifies.

NoteWhat does this look like in JavaScript & React? 
// JavaScript Method Chaining (Data Handling)
const activeUsers = users
    .filter(user => user.isActive)
    .sort((a, b) => a.id - b.id);

// React UI Equivalent (Applying styles/actions)
// React rarely uses dot-chaining for components; instead, you stack physical props or merge class strings:
<button 
    className="bg-blue-500 text-white font-bold py-2 px-4 rounded"
    onClick={handleTap}
>
    Enter Dashboard
</button>

Swift Equivalent (SwiftUI Modifiers): In SwiftUI, you strictly apply styles and behaviors using “View Modifiers”, which are just functions chained vertically via dot notation.

// Enterprise App Snippet: Structuring a styled button
Button(action: handleTap) {
    Text("Enter Dashboard")
}
.font(.headline)
.fontWeight(.bold)
.foregroundColor(.white)
.padding(.vertical, 12)
.padding(.horizontal, 24)
.background(Color.blue)
.cornerRadius(8)
