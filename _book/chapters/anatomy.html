Anatomy of a Basic App

The Folder Structure

Every iOS project natively uses Xcode as the development environment. The folder structure relies on specific files existing in order for the application to compile correctly.

High-Level Xcode Structure:

  • App.swift — The root entry point of the app.
  • ContentView.swift — The first screen the user actually sees.
  • Assets.xcassets — Where you store images, custom colors, and app icons.
  • Info.plist — Configuration file containing hardware permissions and app rules.
NoteWhat does this look like in React?

React projects are more flexible, but generally map similarly:

  • index.js (Root Entry)
  • App.js (Main Router/Screen)
  • public/ or src/assets/ (Images/Icons)
  • package.json (Configuration)

The Root Entry Point

When a user taps an app icon on their iPhone, iOS looks for the @main attribute to know strictly where the app officially starts executing.

Here is the Swift Entry Point (App.swift):

import SwiftUI

@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView() // ⬅️ Points to the first Screen
        }
    }
}
NoteReact Equivalent (index.js) 
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';

const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(
  <React.StrictMode>
    <App /> {/* ⬅️ Points to the first Screen */}
  </React.StrictMode>
);

What About main.swift?

Before @main was introduced in Swift 5.3 (2020), the conventional entry point was a special file named main.swift. It is the only file in a Swift project that is allowed to contain top-level executable code — meaning code written directly at the file’s root scope, outside of any struct, class, or func.

// main.swift (legacy entry point)
import UIKit

UIApplicationMain(
    CommandLine.argc,
    CommandLine.unsafeArgv,
    nil,
    NSStringFromClass(AppDelegate.self)
)

The name main.swift is not arbitrary — the Swift compiler treats it as a reserved filename with special rules:

  • Only one main.swift can exist per target. Having more than one will cause a compile error.
  • Top-level expressions are only legal here. Any other .swift file that tries to run code at the top level (outside a type or function) will fail to compile.
  • @main replaced it. Modern SwiftUI apps use @main on a struct instead, which is cleaner and avoids manually calling UIApplicationMain. You will still encounter main.swift in older UIKit projects or Swift command-line tools.
NoteNode.js Parallel

main.swift behaves much like a Node.js entry file (e.g. index.js run directly via node index.js) — it is the one file where code runs immediately at the top level without needing to be wrapped in a function or class. All other files must export or be explicitly called.

How Screens Work (Views)

In Swift, there is no generic <div>. A screen is always built strictly as a Struct conforming to the View protocol.

Here is what ContentView.swift looks like.

import SwiftUI

struct ContentView: View {
    // ⬇️ Q3: Where is the data? 
    // Data is held explicitly inside the struct instance using Property Wrappers like @State.
    @State private var userName: String = "Charlotte" 

    // ⬇️ Q2: When does render happen? 
    // ANY time the property marked as @State explicitly changes, the 'body' automatically re-renders.
    var body: some View {
        VStack {
            Text("Hello, \(userName)")
            
            Button("Change Name") {
                // ⬇️ Q1: Is this single-threaded?
                // Yes. By default, ALL UI code executes strictly on the Main Thread.
                userName = "Swift Developer"
            }
        }
    }
}

Q1. Is this single-threaded?

  • UI rendering in Swift is purely single-threaded.
  • All interface rendering runs on what Apple calls the Main Thread.
  • In the code block above, when you tap the button, that line of code immediately executes on the Main Thread to guarantee a buttery-smooth 60 to 120 frames per second response.
  • Rule: If you fetch data from the internet (which is slow), you must explicitly push that task to a “Background Thread” to avoid freezing the screen.

Q2. When does the render happen?

Rendering is reactive, triggered purely by a change in State.

  • When the button is tapped and userName changes from “Charlotte” to “Swift Developer”, SwiftUI detects the change in @State and immediately invalidates the current body.
  • SwiftUI then re-computes the physical screen to match the updated state. This is highly optimized and functionally matching how React triggers a re-render when you call setState().

Q3. Where is the data stored?

Fact: Local state data is stored specifically inside the Struct instance using Property Wrappers.

  • Structs are highly efficient, temporary data structures.
  • The data (userName) only exists for as long as this specific screen is visually alive on the device.
  • If you have global data (state you want to share across multiple screens), you store it in classes called ObservableObjects, which act very much like React Context or an overarching Redux Store.
