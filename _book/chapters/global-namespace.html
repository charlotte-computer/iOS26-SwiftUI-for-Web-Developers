Global Namespace

In JavaScript, there is always a global object — window in the browser, global in Node.js. Any variable you declare with var at the top level automatically becomes a property on that object, and you can inspect or mutate the global scope at runtime.

Swift has no such object. There is no window, no global, nothing to inspect. Instead, the global namespace in Swift is your module — the compiled unit of code that makes up your app or framework. It is a flat, statically-typed scope that the compiler fully resolves before your app ever runs.

Namespaces

A namespace is a named region of code. In Swift, any type that contains nested types creates its own namespace. Nested names cannot be accessed without the dot operator .:

class Dog {             // Dog is a namespace (because it contains a nested class)
    class Annie {}      // Annie is a nested type
}
   
Dog.Annie               // ✅ Correct — must go through Dog
Annie                   // ❌ Compile error — Annie does not exist at the top level
NoteReact Parallel

This is similar to named exports. If Annie is a named export inside a Dog module, you cannot use Annie directly without importing or destructuring it from Dog first.

import { Annie } from './Dog'  // explicit — similar to Dog.Annie

Modules

Every Swift target (your app, a Swift package, a framework) compiles into a module. The module is itself a namespace. So the fully qualified name of Annie above is actually:

MyApp.Dog.Annie

Swift just lets you omit the module prefix when you are already inside that module — the same way you do not have to write MyApp.ContentView from within your own app.

import statements bring an external module’s top-level declarations into scope so you do not have to prefix everything:

import Foundation   // gives you Date, URL, etc. without writing Foundation.Date
import SwiftUI      // gives you View, Text, etc. without writing SwiftUI.View

Without the import, the types still exist — you would just need their full qualified name (Foundation.Date).

NoteReact Parallel 
// Without import — you'd need the full path every time (impractical)
const d = new globalThis.Date()

// With import — short name available directly
import { format } from 'date-fns'

What’s in the Global Namespace?

Anything declared at the top level of a Swift file — meaning outside of any struct, class, enum, or func — lives in the module’s global namespace and is accessible from every other file in the same target without any import:

What you can declare globally Example
Types (struct, class, enum, actor, protocol) struct User { }
Global variables and constants let appVersion = "1.0"
Top-level functions func greet() { }
Extensions extension String { }
Operators func +(lhs: Point, rhs: Point) -> Point { }
typealias typealias UserID = String

By default, these are internal — visible to every file in your module, but not visible to other modules (other frameworks or apps that might import yours). You can widen this to public if you are building a library.

NoteReact Parallel — No Direct Equivalent

JavaScript modules require explicit exports. Swift’s internal is more like writing a variable in a shared module store — everything in the same app can see it without you doing anything extra.

// JS — you must export explicitly
export const appVersion = "1.0"

// Swift — no export needed within the same module
let appVersion = "1.0"  // every file in the app can use this

How to Write to the Global Namespace

Declare something at the top level of any .swift file (outside all types):

// AppConstants.swift

let appVersion: String = "2.0"         // global constant
var currentEnvironment: String = "prod" // global variable (mutatable)

func logEvent(_ name: String) {         // global function
    print("[\(currentEnvironment)] \(name)")
}

Every other file in the app can now use appVersion, currentEnvironment, and logEvent() with no import needed.

The Recommended Pattern — Use a Namespace Type

Raw global variables work, but they pollute the flat module scope and can cause naming collisions as the codebase grows. The Swift community convention is to wrap related globals inside a struct or a caseless enum with static members:

// AppConstants.swift

enum App {                              // caseless enum — cannot be instantiated
    static let version = "2.0"
    static var environment = "prod"
    
    enum API {                          // nested namespace
        static let baseURL = "https://api.example.com"
        static let timeout: Double = 30
    }
}

Usage from anywhere in the app:

print(App.version)          // "2.0"
print(App.API.baseURL)      // "https://api.example.com"
App.environment = "staging"

A caseless enum is preferred over struct here because you cannot accidentally create an instance of it (App() would be a compile error), making it a pure namespace with no ambiguity.

NoteReact Parallel — const config Object

This pattern maps closely to a shared config object in JavaScript:

// constants.js
export const App = {
  version: "2.0",
  environment: "prod",
  API: {
    baseURL: "https://api.example.com",
    timeout: 30,
  },
}

// usage
import { App } from './constants'
console.log(App.API.baseURL)

The key difference: Swift’s enum namespace is enforced at compile time — you cannot pass it around as a value, extend it unexpectedly, or accidentally mutate it without static var.
